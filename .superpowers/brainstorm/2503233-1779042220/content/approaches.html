Implementation Approach

How should the nav position switching be built?

A

CSS-only class on root (simple)

Store position in localStorage. Apply data-nav-position="bottom" to <html>. Write CSS rules for each variant. No changes to component logic.

Pros

  • Minimal JS changes
  • Fast toggle (no re-render)

Cons

  • Current sidebar markup isn't designed for 4 positions — CSS hacks pile up
  • Hard to maintain as nav grows
B

Single adaptive Sidebar component (recommended)

Extend Sidebar.tsx with a position prop. The root layout flexes around it. A useNavPosition hook reads/writes the preference (localStorage + optional DB sync). Settings page and right-click context menu both call the same hook.

Pros

  • Single source of truth
  • Clean prop API
  • Easy to extend to new positions

Cons

  • Sidebar component gets 4 rendering branches
C

Separate layout components per position (explicit)

Create SidebarLeft, SidebarRight, NavTop, NavBottom as distinct components. Root layout swaps which one is rendered based on the preference.

Pros

  • Each variant is self-contained and easy to reason about

Cons

  • Lots of duplication — same nav links, icons, theme toggle, notifications in 4 places
  • Bug in one = fix in all

I recommend B — one component, position-aware. Clean and maintainable.
